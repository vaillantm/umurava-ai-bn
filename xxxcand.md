## Job-Selected Upload And Screening Flow

This is the flow the frontend should follow when the recruiter has already selected a job.

### 1. Select job in the dashboard

- The frontend already stores the chosen job.
- Use that selected `jobId` for every upload and screening request.
- Do not let uploads happen without a selected job.

### 2. Upload candidates to that job

Use either of these:

- `POST /api/uploads/json`
- `POST /api/uploads/bulk-pdf`

For JSON upload, send:

```json
{
  "jobId": "69eaa6673f8466145118a14b",
  "candidates": [
```

### 3. What gets saved in the backend

For each candidate the backend saves:

- `Candidate` record
- `Application` record linked to the selected `jobId`

This is the important link:

- `Candidate` = person profile and parsed resume data
- `Application` = candidate applied to this specific job

### 4. Screen after upload

After the upload succeeds, run:

- `POST /api/screenings/run`

Example body:

```json
{
  "jobId": "69eaa6673f8466145118a14b",
  "shortlistSize": 10
}
```

This will:

- read all applications for that job
- score candidates
- create a `Screening` record
- update application statuses
- update shortlisted candidates on the job

### 5. Refresh dashboard data

After upload or screening, the frontend should refetch:

- `GET /api/dashboard/snapshot`
- `GET /api/screenings/latest?jobId=69eaa6673f8466145118a14b`

### 6. Frontend rules

- If no job is selected, disable upload buttons.
- Show the selected job title in the upload panel.
- Attach `jobId` automatically to every upload request.
- After success, refresh the job summary, candidate list, and latest screening panel.

### 7. Suggested dashboard state flow

- user selects job
- `selectedJobId` is stored in state
- user uploads candidate JSON
- frontend sends `jobId` + candidate data
- backend stores candidates and applications
- frontend runs screening
- frontend reloads dashboard snapshot and screening
